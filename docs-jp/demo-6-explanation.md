# Demo 6: statusでフィルタリング（低カーディナリティ）

## クエリ

```sql
SELECT COUNT(*) FROM orders WHERE status = 'pending';
```

## 何が起こるか

statusの値は5種類（`pending`、`processing`、`shipped`、`delivered`、`cancelled`）しかなく、それぞれが100万行のうち約20%ずつ分布しています。つまり `WHERE status = 'pending'` は約200,000行にマッチします。

`status` にインデックスを作成すると、わずかな高速化（約1.3〜1.5倍）が見られるかもしれませんが、Demo 2でインデックスが600倍以上の高速化をもたらしたケースと比較してみてください。インデックスは多少の効果がありますが、いずれにせよ200,000行を処理する必要があるため、改善はわずかです。

## なぜ改善がわずかなのか

これは**選択性**の問題です — フィルタにマッチするテーブルの割合がどれくらいかということです。

Demo 1〜4では、クエリがテーブルのごく一部（0.002%〜0.5%）にマッチしたため、インデックスによってほぼすべてをスキップできました。ここではテーブルの20%がマッチします。インデックスによって残りの80%の読み取りを避けられますが、それでも200,000行 — テーブル全体の5分の1 — にアクセスしています。

**実際に何が起こるか**: PostgreSQLは**Index Only Scan**を使う場合があります — `COUNT(*)` はどの行がマッチするかだけを知ればよく、行の全内容は不要なため、テーブルにアクセスせずインデックスだけで回答できます。これはインデックスがテーブルより小さいため、完全なSequential Scanよりわずかに速くなります。しかし「わずかに速い」がポイントです — 大きな構造の全体をスキャンする代わりに、小さな構造の5分の1をスキャンしているに過ぎません。（正確な実行計画はテーブル統計情報やvisibility mapの状態によって異なり、PostgreSQLはBitmap Index ScanやSeq Scanを選択する場合もあります。）

クエリが `SELECT * FROM orders WHERE status = 'pending'`（全カラムが必要）の場合、結果はさらに悪くなります。PostgreSQLはランダムI/Oでテーブルから200,000行を読み取る必要があり、テーブル全体を順次スキャンするより*遅く*なる可能性があります。その場合、PostgreSQLはインデックスを完全に無視してSeq Scanを行うことがよくあります。

## 重要なポイント

インデックスの価値は、どれだけの作業をスキップできるかに比例します。フィルタがテーブルの20%にマッチする場合、インデックスがスキップできるのは80%の作業だけです。これはせいぜいわずかな改善です — 1.3倍の高速化は、インデックスによって時間の23%を節約したことを意味します。Demo 2ではインデックスが99.998%の行をスキップできたことと比較してみてください。

## 経験則

カラムのインデックスが最も効果的なのは、クエリがごく少数の行を選択する場合 — 通常5〜10%未満のときです。それを超えると改善はわずかになります。この閾値を超えれば超えるほど、インデックスの効果は小さくなります。

**低カーディナリティ**（異なる値の数が少ないこと）は通常、低い選択性を意味し、インデックスの効果が最小限になることを意味します。よくある例:
- `status`（pending/active/closed）
- `is_active`（true/false）
- `type`（少数のカテゴリ）
- `country`（ほとんどのユーザーが1つの国に集中している場合）

## 効果がある場合

例外もあります。分布が偏っている場合 — たとえば注文の95%が `delivered` で、`cancelled` は1%しかない場合 — インデックスは `WHERE status = 'cancelled'`（非常に選択性が高い）に対しては大きな効果がありますが、`WHERE status = 'delivered'`（選択性がまったく高くない）に対しては効果がありません。PostgreSQLはカラム統計情報を使って値ごとにこの判断を行えるほど賢くできています。

## 実務的な判断

このインデックスを追加すべきでしょうか？それはコンテキストによります:
- このクエリが毎秒何千回も実行される場合、一貫した1.3倍の改善は合計すると大きな意味を持ちます。
- たまにしか実行されない場合、すべてのINSERT/UPDATEでインデックスを維持するための書き込みオーバーヘッドは、わずかな読み取り改善に見合わない可能性があります。
- インデックスはディスク容量も消費します — その20〜30%の高速化がストレージと書き込みのペナルティに見合うかどうかの問題です。

## クリーンアップ

