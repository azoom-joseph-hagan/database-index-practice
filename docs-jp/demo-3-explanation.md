# Demo 3: orders.created_at に対する日付範囲検索

## クエリ

```sql
SELECT * FROM orders WHERE created_at BETWEEN '2024-06-01' AND '2024-06-07';
```

## インデックスがない場合の動作

PostgreSQL は1,000,000行すべてを読み取り、各行の `created_at` が7日間のウィンドウ内に収まるかどうかを確認します。注文データは約3年間（2022年〜2024年）にわたるため、1週間分はデータ全体の約0.6%に過ぎません。しかしインデックスがなければ、PostgreSQL はそれらの行がどこにあるか分からないため、すべてを確認する必要があります。

## インデックスが効果的な理由

```sql
CREATE INDEX idx_orders_created_at ON orders (created_at);
```

タイムスタンプカラムに対する B-tree インデックスは、値を時系列でソートして保持します。PostgreSQL はツリー内で `2024-06-01` の位置に直接ジャンプし、`2024-06-07` を超えるまでインデックスを前方に走査します。その狭い範囲の行だけを読み取ります。

EXPLAIN の出力では、Seq Scan が **Bitmap Index Scan** に置き換わり、`Index Cond` に範囲の両端が表示されていることが確認できます。

## インデックスの良い候補である理由

- **範囲クエリ**: B-tree インデックスはデータがソートされているため、`BETWEEN`、`>`、`<`、`>=`、`<=` を効率的に処理できます。PostgreSQL は開始位置を見つけて前方にスキャンするだけで済み、すべての行を確認する必要がありません。
- **広い範囲に対する狭いウィンドウ**: 約1,095日間のうちの7日間は、データ全体の1%未満です。全体の広がりに対して範囲が狭いほど、インデックスの効果は大きくなります。
- **よくあるパターン**: 日付範囲によるフィルタリングは、実際のアプリケーションで最も頻繁に行われる操作の1つです。日付で注文を絞り込む、最近のアクティビティを検索する、特定期間のレポートを生成するなど、さまざまな場面で使われます。フィルタリングに使用する日付カラムにはほぼ常にインデックスを作成する価値があります。

## クリーンアップ

```sql
DROP INDEX idx_orders_created_at;
```
