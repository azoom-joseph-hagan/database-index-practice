# インデックスの基礎

## インデックスとは何か？

データベースのインデックスは、教科書の巻末にある索引のようなものです。500ページの本の中から「PostgreSQL」という言葉が出てくる箇所をすべて見つけたい場合、2つの方法があります。

1. **すべてのページを読む** — 確実にすべて見つかりますが、非常に時間がかかります。
2. **索引を確認する** — 「PostgreSQL: 12, 47, 203ページ」と書いてあるので、そこに直接飛べます。

データベースのインデックスもまったく同じ仕組みです。インデックスがなければ、PostgreSQLは目的のデータを見つけるためにテーブルのすべての行を読まなければなりません（**Sequential Scan**）。インデックスがあれば、該当する行に直接ジャンプできます。

## インデックスの仕組み（簡略版）

あるカラム — 例えば `last_name` — にインデックスを作成すると、PostgreSQLは値から行の位置へのマッピングを持つ、ソート済みの独立したデータ構造（通常は**B-tree**）を構築します。

```
"Adams"   → row 4821
"Adams"   → row 31002
"Brown"   → row 156
"Brown"   → row 8934
"Chen"    → row 2201
...
```

B-treeは、各ノードが数百の子ノードを持てるバランス木です。これにより、PostgreSQLは10億行のテーブルでも、ツリーをわずか3〜4回たどるだけで任意の値を見つけることができます。すべての行をチェックする代わりに、ほぼ瞬時に正確な位置まで絞り込めるのです。

> **補足:** PostgreSQLはB-tree以外にも、全文検索や地理空間データなどの特殊な用途向けに他のインデックスタイプ（Hash、GIN、GiST、BRIN、SP-GiST）をサポートしています。B-treeがデフォルトであり、大多数のユースケースをカバーするため、ここではB-treeに焦点を当てます。

## インデックスが有効な場合

インデックスは、クエリの**選択性**が高い場合に最も効果を発揮します。つまり、テーブル全体のうちごくわずかな行だけを返すクエリです。

- 50,000件の中から特定の顧客を名前で検索する場合
- 1,000,000件の中からある顧客の20件の注文を見つける場合
- 3年分の日付データから7日間のウィンドウでフィルタリングする場合

インデックスによってスキップできる行が多いほど、高速化の効果は大きくなります。

## インデックスが有効でない場合

インデックスは万能ではありません。以下の場合、インデックスは無意味、あるいはむしろ有害です。

- **選択性が低い場合**: フィルタがテーブルの20%以上にマッチする場合（例えば、5種類しかない `status` でフィルタリングする場合）、PostgreSQLはインデックスとデータページの間を行ったり来たりするよりも、テーブル全体をシーケンシャルに読んだ方が速いと判断します。
- **テーブルが極めて小さい場合**: テーブルが500行しかなければ、数ページのディスクに収まります。全体を読んでも十分高速なので、インデックスによる意味のある改善はありません。
- **どのみちすべての行が必要な場合**: `COUNT(*) GROUP BY status` のようなクエリは、いずれにせよすべての行を処理する必要があります。インデックスでスキップできるものはありません。


## インデックスのコスト

インデックスにはコストがあります。
- ディスク容量を消費します（インデックス対象カラムのデータをソートした別のコピーが作られます）。
- すべての `INSERT`、`UPDATE`、`DELETE` は関連するインデックスもすべて更新する必要があるため、書き込みが遅くなります。
- **VACUUMのオーバーヘッド**が増加します。PostgreSQLのVACUUMプロセス（削除・更新された行の領域を回収するバックグラウンドクリーナー）は、テーブル上のすべてのインデックスの不要なエントリもクリーンアップしなければなりません。インデックスが多いほどVACUUMに時間がかかり、VACUUMが追いつかなくなるとインデックスの肥大化が発生します。
- 使われていないインデックスは純粋なオーバーヘッドです。書き込み、ストレージ、VACUUMの時間がかかるだけで、読み取りの改善はゼロです。

だからこそ、すべてのカラムにインデックスを作るべきではありません。パフォーマンスが重要なクエリで実際にフィルタ、結合、ソートに使うカラムにのみインデックスを作成しましょう。

# EXPLAIN ANALYZEの読み方

`EXPLAIN ANALYZE` は、PostgreSQLがクエリをどのように実行したかを正確に表示する機能です。どの戦略を選択し、各ステップにどれだけの時間がかかったかがわかります。クエリのパフォーマンスを理解するための最も有用なツールの一つです。

## 実行方法

```sql
EXPLAIN ANALYZE SELECT * FROM orders WHERE customer_id = 42;
```

`EXPLAIN` の部分は実行プランを表示します。`ANALYZE` の部分は実際にクエリを実行し、実測のタイミング数値を追加します。

## 出力の読み方

典型的な出力は以下のようになります。

```
Seq Scan on orders  (cost=0.00..20457.00 rows=21 width=31) (actual time=0.03..85.12 rows=24 loops=1)
  Filter: (customer_id = 42)
  Rows Removed by Filter: 999976
Planning Time: 0.05 ms
Execution Time: 85.15 ms
```

各要素を分解して見ていきましょう。

### スキャンタイプ

最初の単語は、PostgreSQLがデータにどのようにアクセスしたかを示します。

| スキャンタイプ | 意味 | 表示される状況 |
|-----------|---------------|-----------------|
| **Seq Scan** | テーブルのすべての行を1行ずつ読む | 有用なインデックスがない、またはインデックスが役立つほどクエリの選択性が高くない場合 |
| **Index Scan** | インデックス経由で行を検索し、完全な行データを取得する | 選択性が高いクエリで、適切なインデックスがある場合 |
| **Bitmap Index Scan** + **Bitmap Heap Scan** | まずインデックスをスキャンして該当行の「マップ」を構築し、それからまとめて読み取る | Index Scan（各行がランダムI/Oのホップになる）には多すぎるが、テーブル全体を読むのは無駄になるほど少ない行数の場合。目安：大きなテーブルから数百〜数千行程度。 |
| **Index Only Scan** | テーブルにアクセスせず、インデックスだけでクエリに回答する | クエリが必要とするすべてのカラムがインデックスに含まれている場合。**注意点:** テーブルに最近多くの書き込みがあり、VACUUMが最近実行されていない場合、PostgreSQLは行の可視性を確認するためにテーブルをチェックする必要があるかもしれません。`Heap Fetches` の数値が高い場合は、テーブルへのフォールバックが発生していることを意味します。 |

### コスト見積もり

```
(cost=0.00..20457.00 rows=21 width=31)
```

- `cost=0.00..20457.00` — PostgreSQLが行の返却を開始するまでの*推定*コスト（`0.00`）と合計コスト（`20457.00`）。これはミリ秒ではなく、PostgreSQLのコストモデル（`seq_page_cost` や `random_page_cost` などの設定に基づく）から算出される内部単位です。プラン同士を比較するのには有用ですが、絶対的な時間としては使えません。
- `rows=21` — PostgreSQLが見つかると*推定*した行数。
- `width=31` — 行の平均サイズ（バイト単位）。

### 実測タイミング

```
(actual time=0.03..85.12 rows=24 loops=1)
```

- `actual time=0.03..85.12` — 実際のミリ秒。最初の行を返すまでの時間（`0.03`）とすべての行を返すまでの時間（`85.12`）。
- `rows=24` — *実際に*見つかった行数（上記の推定値と比較してみてください）。
- `loops=1` — このステップが何回実行されたか。Nested Loopやパラレルクエリでは、1より大きくなることがあります。

### フィルタ行

```
Filter: (customer_id = 42)
Rows Removed by Filter: 999976
```

これは、PostgreSQLが行を読んだ後に、マッチしなかった行を破棄したことを示しています。Seq Scanで `Rows Removed by Filter` の数値が高い場合は、インデックスが有効であるという強いシグナルです。PostgreSQLがわずか24行を見つけるために約100万行をチェックしたことを意味しています。

### ネストされたプラン

プランはツリー構造で表示されます。インデントされた行は子操作です。

```
Nested Loop  (cost=4.59..516.91 rows=63 width=24) (actual time=0.04..1.27 rows=68 loops=1)
  ->  Bitmap Heap Scan on orders  (cost=4.59..85.61 rows=21 width=10) (actual time=0.03..0.21 rows=24 loops=1)
        Recheck Cond: (customer_id = 37760)
        ->  Bitmap Index Scan on idx_orders_customer_id  ...
  ->  Index Scan using idx_order_items_order_id on order_items  (cost=0.43..20.50 rows=4 width=18) (actual time=0.03..0.04 rows=3 loops=24)
```

下から上へ読みましょう。最も内側の操作が最初に実行されます。ここでは、PostgreSQLは以下の順序で処理しています。
1. インデックスを使って該当するorder IDを見つける
2. それらのorder行をフェッチする
3. 各orderについて、別のインデックスを使ってorder_itemsを見つける
4. Nested Loopで結合する



## 単一クエリの実行時間が誤解を招く理由

単独で見ると15msのクエリは問題ないように見えます。しかし本番環境では、2つの要因によって状況は大幅に悪化します。

### ネットワークレイテンシ

これらのデモはローカルで実行されます。アプリケーションとデータベースが同じマシン上にあるため、ネットワーク遅延はほぼありません。本番環境では、データベースは通常別のサーバーにあります。各クエリにつき、**ネットワークラウンドトリップ**が1〜5ms（同一データセンター）、または20〜50ms以上（異なるリージョン）追加されます。

クエリごとに5msのネットワークオーバーヘッドが加わる場合：

- **インデックスありのクエリ**: 0.02ms（DB処理）+ 5ms（ネットワーク）= **合計約5ms**
- **インデックスなしのクエリ**: 15ms（DB処理）+ 5ms（ネットワーク）= **合計約20ms**

合計時間の差は小さく見えます（4倍）。しかし、インデックスなしのクエリではデータベースが**750倍の仕事量**をこなしています。そして、多数のリクエストが同時に到着すると、その仕事量は急速に蓄積されます。

### 負荷時の累積コスト

データベースは一度に1つのリクエストだけを処理するわけではありません。100件のリクエストがあるエンドポイントに到達した場合：

- **インデックスなし**: 15ms × 100 = **1.5秒**のデータベース処理時間
- **インデックスあり**: 0.02ms × 100 = **2ミリ秒**

遅いクエリはデータベース接続をより長く保持します。ほとんどのアプリケーションは10〜20個のコネクションプールを使用しています。遅いクエリでそれらが占有されると、新しいリクエストはキューに入り、ユーザーにはタイムアウトが表示されます。インデックスありのクエリはほぼ即座にコネクションを解放するため、プールの空きが保たれます。

デモではこれを直接確認できます。各EXPLAIN ANALYZEの後に「100件のリクエストがこのエンドポイントに到達した場合」のセクションがあり、実際の累積ウォールクロック時間を表示しています。デモ4（JOINクエリ）が最も劇的です。インデックスなしで約7秒、インデックスありで約40msです。

インデックスの欠如は開発環境では見えませんが、本番トラフィック下ではアプリケーション全体のパフォーマンスを劣化させます。

## クイックリファレンス

| 表示内容 | 意味 | 対処法 |
|-------------|---------------|------------|
| `Seq Scan` + 高い `Rows Removed by Filter` | テーブル全体をスキャンし、大部分の行を破棄している | フィルタ対象カラムにインデックスを追加する |
| 小さなテーブルでの `Seq Scan` | テーブル全体をスキャンしているが、テーブルが極めて小さい | 対処不要 — 既に十分高速 |
| `Seq Scan` + 低選択性フィルタ | テーブル全体をスキャンしているが、大部分の行がマッチしている | 対処不要 — インデックスは役に立たない |
| `Index Scan` または `Bitmap Heap Scan` | インデックスを効果的に使用している | 対処不要 |
| 推定 `rows` と実際の `rows` に大きな乖離 | 統計情報が古くなっている | `ANALYZE tablename` を実行する |
| `Nested Loop` で内部テーブルに `Seq Scan` | 結合の中でテーブル全体のスキャンが繰り返されている | 結合カラムにインデックスを追加する |
