# Demo 2: customer_id による orders の検索（1,000,000行）

## クエリ

```sql
-- customer_id は実行ごとに異なります。ここでは例として 37760 を使用します
SELECT * FROM orders WHERE customer_id = 37760;
```

## インデックスがない場合の動作

PostgreSQL は、この顧客に属する約20行を見つけるために、`orders` テーブルの1,000,000行すべてをスキャンしなければなりません。並列ワーカーが処理を分担しても、約100万行を読み込んで破棄していることに変わりありません。

EXPLAIN の出力では以下のような結果が表示されます:

```
Parallel Seq Scan on orders
  Filter: (customer_id = 37760)
  Rows Removed by Filter: 333325   ← ワーカーごとの値なので、合計で約1M行
```

## インデックスが効果的な理由

```sql
CREATE INDEX idx_orders_customer_id ON orders (customer_id);
```

これは外部キーインデックスの典型的なユースケースです。`customer_id` カラムは `customers` テーブルを参照しており、「顧客Xのすべての注文を取得する」という問い合わせを頻繁に行います。

インデックスがあれば、PostgreSQL は B-tree で `customer_id = 37760` を参照し、一致する約20行の位置を特定して、それらの行だけを取得します。1M行のスキャンが約20行の読み取りに変わります。

## インデックスの良い候補である理由

- **極めて高い選択性**: 1,000,000行中わずか20行 — テーブル全体の0.002%です。
- **大規模テーブル**: 1M行では、フルスキャンとピンポイント検索の差は非常に大きくなります。
- **外部キーパターン**: インデックスを追加する最も一般的な理由の1つです。PostgreSQL は主キーとは異なり、外部キーカラムにインデックスを自動作成*しません*。FK制約を定義してもインデックスを追加しなければ、そのFKカラムによるすべての検索はフルテーブルスキャンになります。

これはデータベースに追加できるインデックスの中で最も重要なものであり、かつ最も忘れられがちなものの1つです。

## クリーンアップ

```sql
DROP INDEX idx_orders_customer_id;
```
