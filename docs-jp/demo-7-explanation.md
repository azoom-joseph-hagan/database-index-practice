# Demo 7: COUNT(*) GROUP BY status

## クエリ

```sql
SELECT status, COUNT(*) FROM orders GROUP BY status;
```

## 何が起こるか

このクエリは各statusの値ごとに注文数を集計する必要があります。`WHERE` 句がないため、正確な集計結果を得るにはすべての行を参照する必要があります。

`status` にインデックスがあっても、パフォーマンスは実質的に変わりません。

## なぜインデックスが役に立たないのか

インデックスは行を*スキップ*するためのものです。しかしこのクエリでは何もスキップできません — 正確な合計を得るにはテーブル内のすべての行をカウントしなければならないのです。フィルタで除外できる行が存在しません。

教科書の例えを使うと: 「この本に各章が何回登場するか」と聞かれた場合、巻末の索引は役に立ちません — いずれにせよすべてのページ参照を確認する必要があります。最初から最後まで本を読む方が単純です。

具体的には:

- **インデックスなし**: PostgreSQLはSequential Scanを行い、100万行すべてを読み取り、グループ化してカウントします。並列ワーカーを使って処理を分割することもできます。
- **インデックスあり**: PostgreSQLには2つの選択肢がありますが、どちらも優れているわけではありません:
  1. インデックスを無視して同じSequential Scanを行う（通常はこちらが選ばれます）。
  2. Index Only Scanを行う — テーブルの代わりにインデックスから100万エントリすべてを読み取る。インデックスはテーブル全体より小さいため*わずかに*速くなる可能性がありますが、すべてのエントリを読み取ることに変わりないため、改善はわずかです。

## まとめ

すべての行にアクセスする必要がある集約クエリ（選択的な `WHERE` を伴わない `COUNT`、`SUM`、`AVG`、`GROUP BY`）は、インデックスから意味のある恩恵を受けません。ボトルネックは処理すべきデータの総量であり、インデックスはそれを減らすことができません。

これらの集約を高速にする必要がある場合、解決策はインデックスとは異なるアプローチになります:
- **マテリアライズドビュー**: 集約を事前に計算し、定期的にリフレッシュする。
- **サマリーテーブル**: データの変更に合わせて集計値を逐次更新する。
- **パーティショニング**: テーブルを日付やstatusで分割し、データベースが関連するパーティションのみをスキャンするようにする。
