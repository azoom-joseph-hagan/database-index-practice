# Demo 9: WHERE total_amount > 0（95%以上がマッチ）

## クエリ

```sql
SELECT COUNT(*) FROM orders WHERE total_amount > 0;
```

## 何が起こるか

ほぼすべての注文の `total_amount` は0より大きくなっています（シードデータは10〜2000の範囲で値を生成します）。つまりこのフィルタは100万行の95〜100%にマッチします。

`total_amount` にインデックスがあっても、PostgreSQLはそれを無視してSequential Scanを行います。インデックスの有無にかかわらずEXPLAIN計画は同一です — PostgreSQLはインデックスの使用を検討すらしません。

## なぜインデックスが役に立たないのか

これはDemo 6と同じ本質的な問題ですが、より顕著です。問題は異なる値の数ではなく、**述語の選択性**です。

`WHERE total_amount > 0` はほぼすべての行にマッチします。仮にPostgreSQLがインデックスを使用したとしても、ほぼ全体をスキャンすることになり、意味のあるスキップができません。その場合、テーブルを直接読み取ることに対するメリットはありません。

Demo 6（選択性20%）では、PostgreSQLは80%の行をスキップできたため、インデックスからわずかながらも効果が得られました。ここでは選択性が約100%であり、スキップできるものが何もありません。プランナーはこれを認識し、インデックスをまったく使用しません — どちらの計画もParallel Seq Scanを使用します。

## 重要なポイント

重要なのはカラムにインデックスがあるかどうかではなく、特定のクエリの値が選択的かどうかです。同じ `total_amount` のインデックスでも、異なるクエリでは効果が変わります:

```sql
-- これは選択的: おそらく注文の0.1%
SELECT COUNT(*) FROM orders WHERE total_amount > 1900;

-- これは選択的でない: 注文の95%以上
SELECT COUNT(*) FROM orders WHERE total_amount > 0;
```

PostgreSQLはフィルタリングする実際の値とカラムのデータ分布統計に基づいて、クエリごとにこの判断を行います。

## PostgreSQLはどうやって判断するのか

テーブルに対して `ANALYZE` を実行すると、PostgreSQLはデータをサンプリングし、各カラムについて統計情報を保存します:
- 最頻値とその出現頻度
- 値の分布のヒストグラム
- 異なる値の数

クエリプランナーはこれらの統計情報を使用して、指定された `WHERE` 句にマッチする行数を推定します。推定値がテーブルの一定割合を超えると、インデックスをスキップします。

これが `ANALYZE` の実行（またはautovacuumに任せること）が重要な理由です — 古い統計情報はPostgreSQLに誤った判断をさせる可能性があります。たとえば、Seq Scanの方が速い場面でインデックスを使用したり、その逆が起こったりします。

## クリーンアップ

```sql
DROP INDEX idx_orders_total_amount;
```
