# Demo 8: 小さなテーブル（500行）

## クエリ

```sql
-- アプローチ1: SQLのWHERE（インデックスなし）
SELECT * FROM products WHERE category = 'Electronics';

-- アプローチ2: 全行を取得し、JavaScriptでフィルタリング
SELECT * FROM products;  -- その後: rows.filter(r => r.category === 'Electronics')

-- アプローチ3: SQLのWHERE（categoryにインデックスあり）
SELECT * FROM products WHERE category = 'Electronics';
```

## 何が起こるか

3つのアプローチすべてがほぼ同じ時間で結果を返します — 通常1ms未満です。インデックスは意味のある違いを生みません。

## なぜインデックスが役に立たないのか

`products` テーブルは500行です。テーブル全体がわずか数ページのディスクページに収まります（PostgreSQLのページサイズは8KB）。数ページを読み取ることは実質的に一瞬です。

テーブルがこの程度の小ささの場合:
- **Sequential Scan**は2〜3ページを読み取れば完了です。
- **Index Scan**はインデックスを読み取り（1〜2ページ）、次にテーブルからマッチする行を取得します（1〜2ページ）。実用上のメリットなく、合計I/Oは同じかそれ以上になります。
- **全行をJavaScriptに取得**してメモリ内でフィルタリングすることも問題ありません。転送・処理するデータ量が非常に少ないためです。

## まとめ

小さなテーブルにインデックスを作成する必要はありません。「念のためにすべてにインデックスを作成する」というのはよくある反応ですが、数百行から数千行程度のテーブルでは、インデックスは書き込みオーバーヘッド（すべてのinsert/updateでインデックスの維持が必要）を追加するだけで、読み取りの改善にはなりません。

「小さい」とはどの程度でしょうか？正確な境界線はありませんが、おおよその目安として:
- **約1,000行未満**: インデックスを作成する価値はほぼありません（主キー以外）。
- **1,000〜10,000行**: 通常は価値がありません。ただし、テーブルがループ内で繰り返しクエリされる場合（例: Nested Loop Joinの内部テーブルとして）は例外です。
- **10,000行以上**: 頻繁にフィルタリングやJOINに使用するカラムにはインデックスの検討を始めるべきです。

これらの閾値は行の幅にも依存します — 5,000行で50カラムのテーブルは、5,000行で3カラムのテーブルより大きくなります。

## JavaScriptフィルタリングの疑問

データベースでフィルタリングするのとアプリケーションコードでフィルタリングするのとどちらが良いかは、よくある質問です。小さなテーブルでは、どちらでも構いません — いずれの場合も、データベースへのネットワークラウンドトリップが最大のコストになる可能性が高いです。大きなテーブルでは、常にSQLでフィルタリングすべきです — 何百万行もネットワーク経由で転送して、アプリケーション側でそのほとんどを破棄するようなことは避けるべきです。
